# Story 7.1 — API Connector — Register, Test and Expose as Agent Tools

**Epic:** 7 — API Connector and Data Import
**Status:** Draft
**Points:** 8

## Description

As a user, I want to register external APIs with their auth configuration and test connectivity, so that the AI agent can call any registered API as a tool.

## Acceptance Criteria

1. Tabela `api_connectors` criada via migration: `id` (uuid PK), `name` (text NOT NULL), `base_url` (text NOT NULL), `auth_type` (text NOT NULL CHECK IN ('none','api_key','bearer','basic','oauth2')), `auth_config` (jsonb NULLABLE — encrypted via pgcrypto), `headers` (jsonb NULLABLE), `is_active` (boolean NOT NULL DEFAULT true), `created_by` (uuid NOT NULL FK profiles(id)), `created_at` (timestamptz), `updated_at` (timestamptz)
2. API CRUD: `GET /api/v1/api-connectors`, `POST /api/v1/api-connectors`, `PUT /api/v1/api-connectors/:id`, `DELETE /api/v1/api-connectors/:id`; owner only para criar/editar/deletar; member pode listar os ativos
3. `POST /api/v1/api-connectors/:id/test` faz um `GET` na `base_url` com as credenciais configuradas e retorna `{ status: number, latency_ms: number, ok: boolean }`; timeout 5s
4. UI: formulario de registro com campos dinamicos baseados em `auth_type` (ex: se api_key → campo "Key Name" e "Key Value"; se bearer → campo "Token"; se basic → "Username" e "Password"); botao "Testar Conexao" com feedback visual (verde/vermelho + latencia)
5. Cada API ativa e automaticamente registrada como tool no agent-harness (via dynamic tool loader em `toolRegistry`) — a tool `call_api_connector` usa o conector correto baseado no ID passado como parametro; `auth_config` nunca e exposto para o Claude
6. `auth_config` armazenado encriptado no banco; chave de encriptacao derivada de env var `CONNECTOR_ENCRYPTION_KEY`

## Scope

### IN
- Migration `api_connectors` com RLS e encriptacao de `auth_config`
- Rotas Hono CRUD e endpoint de teste
- UI: pagina ApiConnectorPage com lista e Sheet de registro
- Integracao com agent-harness: tool `call_api_connector` usa os connectors ativos

### OUT
- OAuth2 completo para connectors (apenas suporte a token Bearer/API Key para MVP; OAuth2 completo pos-MVP)
- Chamadas de metodos alem de GET para o endpoint de teste (apenas health check)
- UI para definir endpoints especificos do conector (apenas base_url no MVP — a tool usa paths passados como argumento)

### Tool: call_api_connector (agent-harness)
```typescript
// packages/agent-harness/src/tools/built-in/call-api-connector.ts
// Input: { connector_id: string, path: string, method: 'GET'|'POST'|'PUT'|'DELETE', body?: unknown }
// Permissao: member pode chamar (read-only); POST/PUT/DELETE requerem owner
// isDestructive: quando method !== 'GET'
// A tool busca o conector do DB (via service layer), aplica auth, faz a request e retorna response
```

### RLS (api_connectors)
```sql
ALTER TABLE public.api_connectors ENABLE ROW LEVEL SECURITY;
-- SELECT: owner vê todos; member vê os ativos
CREATE POLICY "connectors_select" ON public.api_connectors FOR SELECT USING (
  public.get_user_role() = 'owner' OR (auth.uid() IS NOT NULL AND is_active = true)
);
-- INSERT/UPDATE/DELETE: owner only
CREATE POLICY "connectors_write" ON public.api_connectors FOR ALL USING (public.get_user_role() = 'owner');
```
