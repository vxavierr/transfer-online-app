### Diagnóstico pós-build #27

O plugin `@capacitor-community/background-geolocation@1.2.26` é declarado compatível com Capacitor v3–v7. O projeto usa Capacitor `8.2.0`. O `codemagic.yaml` aplica um patch `sed` no `Package.swift` do plugin para forçar aceitação do Capacitor 8, mas isso é cosmético — não corrige a regressão de API.

A issue [Capacitor #8333](https://github.com/ionic-team/capacitor/issues/8333) documenta que o Capacitor 8.0.2 via SPM colocou métodos como `call.getString()`, `call.getBool()`, `call.getDouble()`, `call.reject()`, `call.resolve()` atrás de uma feature flag experimental do Swift (`$NonescapableTypes`). O `Plugin.swift` do plugin usa exatamente esses métodos (linhas 90, 93, 106, 156, 219). Resultado provável: callbacks nunca disparam silenciosamente.

**Sintoma bate com diagnóstico:** GPS retorna zero pontos sem nenhum erro visível.

**Plano B ativado:** trocar para `@capgo/background-geolocation@8.0.32` — fork do mesmo plugin com suporte oficial ao Capacitor 8, mesmo padrão de uso (`start(options, callback)` / `stop()`), MPL-2.0, release de 2026-05-04.

### AC2 — Swap do plugin (condicional: apenas se AC1 confirmar que callback não dispara)

Remover `@capacitor-community/background-geolocation`:
```bash
# No diretório app/
bun remove @capacitor-community/background-geolocation
```

Instalar `@capgo/background-geolocation@8.0.32`:
```bash
bun add @capgo/background-geolocation@8.0.32
bunx cap sync
```

**Adaptação de API — diferença crítica:**

O plugin `@capgo` usa `start(options, callback)` / `stop()` em vez de `addWatcher()` / `removeWatcher()`:

```js
// Plugin anterior (@capacitor-community):
import BackgroundGeolocation from '@capacitor-community/background-geolocation';
const watcherId = await BackgroundGeolocation.addWatcher(options, callback);
await BackgroundGeolocation.removeWatcher({ id: watcherId });

// Novo plugin (@capgo):
import { BackgroundGeolocation } from '@capgo/background-geolocation';
await BackgroundGeolocation.start(options, callback);
await BackgroundGeolocation.stop();
```

O `@capgo` gerencia estado internamente — não retorna um ID de watcher. Adaptar `GeoService.js`:
- `startBackgroundTracking()` → chama `BackgroundGeolocation.start(options, callback)`
- `stopBackgroundTracking()` → chama `BackgroundGeolocation.stop()`
- O parâmetro `watcherId` em `stopBackgroundTracking` se torna irrelevante para iOS (manter assinatura para não quebrar callers Android)

**Parâmetros de `StartOptions` confirmados pela documentação oficial do @capgo:**

```js
await BackgroundGeolocation.start(
  {
    backgroundMessage: 'Necessário para rastreamento durante viagens',  // dispara requestAlwaysAuthorization no iOS
    backgroundTitle: 'Transfer Online',
    requestPermissions: true,
    stale: true,      // permite primeiro fix mais rápido (iOS warm-up)
    distanceFilter: 5,
  },
  (location, error) => {
    if (error) { /* tratar erro */ return; }
    if (location) { callback(location); }
  }
);
```

**Nota sobre `backgroundMessage` no @capgo:** A documentação oficial confirma: `"If the backgroundMessage option is defined, the plugin will provide location updates whether the app is in the background or the foreground."` O comportamento de `requestAlwaysAuthorization` é controlado por este parâmetro — mesmo contrato do plugin anterior (conforme `Plugin.swift:90` do `@capacitor-community`).

Remover também o patch `sed` do `codemagic.yaml` que forçava compatibilidade de versão SPM do plugin antigo:
```yaml
# REMOVER esta linha do codemagic.yaml:
# sed -i '' 's|from: "7.0.0"|"7.0.0"..<"9.0.0"|' \
#   node_modules/@capacitor-community/background-geolocation/Package.swift
```

### AC4 — Preservar fluxo de permissão "Always" e o LocationDisclosureModal

O `LocationDisclosureModal` (`app/src/components/disclosure/LocationDisclosureModal.jsx`) é um popup educacional que aparece **antes** do diálogo nativo de permissão iOS. Já implementado. **Não alterar este componente nem o fluxo que o dispara.**

O `backgroundMessage` passado para `BackgroundGeolocation.start()` (AC2) é o que instrui o plugin a chamar `requestAlwaysAuthorization` em vez de `requestWhenInUseAuthorization`. **Este parâmetro é obrigatório** — sem ele, GPS para silenciosamente em background após ~30s mesmo com `UIBackgroundModes: location` presente no Info.plist.

Fluxo esperado preservado:
1. `LocationDisclosureModal` exibe → usuário clica "Continuar"
2. `requestBackgroundPermission()` é chamado
3. Plugin chama `requestWhenInUseAuthorization` (first step iOS 13+)
4. Usuário concede "Ao usar o app"
5. Plugin faz upgrade para `requestAlwaysAuthorization` (porque `backgroundMessage` está presente)
6. Usuário concede "Sempre" → GPS background ativo

### AC5 — Preservar filtro de acurácia adaptativo da Story 1.2

Manter em `TelemetryTracker.jsx`, no `handlePositionUpdate`:

```js
const isFirstFix = !lastPositionRef.current;
const accuracyThreshold = (Capacitor.getPlatform() === 'ios' && isFirstFix) ? 500 : 100;
if (accuracy && accuracy > accuracyThreshold) return;
```

**Por quê:** CLLocationManager iOS retorna o primeiro fix com acurácia 150–300m antes de "warmar". Descartar com threshold de 100m significa nunca começar a rastrear no iOS.

**Validação Android:** `Capacitor.getPlatform() === 'android'` → condição falsa → threshold permanece 100m. Comportamento Android inalterado.
