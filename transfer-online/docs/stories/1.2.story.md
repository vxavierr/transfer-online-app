## Causa raiz descoberta na pesquisa

A Story 1.1 introduziu o branch `if (!TelemetryForeground)` para detectar iOS. Mas:

```js
// TelemetryForegroundBridge.js:22-24
const TelemetryForeground = Capacitor.isNativePlatform()
  ? registerPlugin('TelemetryForeground')  // ← retorna proxy truthy SEMPRE
  : null;
```

`Capacitor.registerPlugin()` retorna um **Proxy truthy mesmo quando o plugin nativo não existe na plataforma atual**. Então no iOS, `TelemetryForeground` é truthy, e `!TelemetryForeground` avalia para `false` — fazendo o iOS executar o caminho Android (`watchPosition()` foreground-only).

**Consequência:** Toda a infraestrutura `BackgroundGeolocation` adicionada na Story 1.1 nunca foi acionada no iOS. Há semanas o iOS roda com `watchPosition()` foreground-only + filtro de acurácia 100m que descarta primeiro fix iOS (~150-300m) + permissão `whenInUse` (não `always`).

Pesquisa completa em `docs/research/ios-gps-research-runtime.md`, `ios-gps-research-architect.md`, `ios-gps-research-analyst.md`.

## Acceptance Criteria

### AC1 — Seletor de plataforma correto
Substituir `if (!TelemetryForeground)` por `if (Capacitor.getPlatform() === 'ios')` em **dois pontos** de `TelemetryTracker.jsx`:
- Em `startGPSMonitoring()` (linha ~313)
- Em `stopGPSMonitoring()` (linha ~342)

Importar `Capacitor` de `@capacitor/core` no topo do arquivo se ainda não estiver importado.

**Validação:** Em runtime no iOS, `Capacitor.getPlatform()` retorna `'ios'` → branch iOS executa. No Android retorna `'android'` → branch Android executa (sem mudança).

### AC2 — `requestBackgroundPermission` pede "Always" de verdade
Em `GeoService.js:81-110`, o `addWatcher` chamado dentro de `requestBackgroundPermission()` **deve passar `backgroundMessage`** para que o plugin sinalize ao iOS que precisa de `requestAlwaysAuthorization` (não `whenInUse`).

```js
BackgroundGeolocation.addWatcher(
  {
    requestPermissions: true,
    backgroundMessage: 'Necessário para rastreamento durante viagens',  // ← adicionar
    backgroundTitle: 'Transfer Online',                                  // ← adicionar
    stale: true,
    distanceFilter: 999999,
  },
  ...
)
```

**Por quê:** O `Plugin.swift` do `@capacitor-community/background-geolocation` define `background = (backgroundMessage != nil)` na linha 90. Sem `backgroundMessage`, o plugin chama `requestWhenInUseAuthorization` em vez de `requestAlwaysAuthorization`. Como `allowsBackgroundLocationUpdates` requer `Always`, o GPS para silenciosamente em background.

### AC3 — Filtro de acurácia adaptativo no iOS
Em `TelemetryTracker.jsx`, no `handlePositionUpdate`, modificar o filtro:

**Antes:**
```js
if (accuracy && accuracy > 100) return;
```

**Depois:**
```js
const isFirstFix = !lastPositionRef.current;
const accuracyThreshold = (Capacitor.getPlatform() === 'ios' && isFirstFix) ? 500 : 100;
if (accuracy && accuracy > accuracyThreshold) return;
```

**Por quê:** O CLLocationManager iOS frequentemente retorna o primeiro fix com acurácia 150-300m antes de "warmar" — descartar isso significa nunca começar a rastrear no iOS. Após o primeiro fix, volta ao threshold normal de 100m.

**Validação Android:** No Android, `Capacitor.getPlatform() === 'android'`, então o threshold permanece 100m em todos os casos. Comportamento atual preservado.
