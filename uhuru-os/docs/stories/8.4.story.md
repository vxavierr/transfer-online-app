# Story 8.4 — Agent Hooks — Permissions, Rate Limiting and Audit

**Epic:** 8 — AI Agent (Harness, Tools, Chat)
**Status:** Draft
**Points:** 8

## Description

As a platform operator, I want the agent to respect user permissions, rate limits, and full audit logging, so that the platform is secure and usage is traceable.

## Acceptance Criteria

1. Pre-tool hook `preToolHook.check(toolUse, ctx)`: verifica se o `ctx.user.role` atende o `tool.permissions.requiredRole`; rejeita com `{ denied: true, reason: 'Insufficient permissions' }` se nao atender
2. Pre-tool hook: rate limiting — verifica se usuario excedeu 30 mensagens/minuto (contador em memoria ou Redis-lite); rejeita com `{ denied: true, reason: 'Rate limit exceeded' }` se excedido; retorna headers `X-RateLimit-Remaining` na response do chat
3. Pre-tool hook: tools com `isDestructive: true` retornam `{ needsConfirmation: true }` em vez de executar; o engine gera evento SSE `confirmation_required` e aguarda proxima mensagem
4. Post-tool hook `postToolHook.log(...)`: grava em `agent_audit_logs` os campos: `session_id`, `tool_name`, `input` (jsonb — sanitizado para remover tokens/credenciais), `output` (jsonb), `duration_ms`, `user_id`, `created_at`; retencao de 90 dias (NFR7)
5. JWT do usuario logado propagado em toda a cadeia: Hono recebe JWT → injeta em `ctx.supabase` → todas as queries do engine e das tools passam pelo mesmo cliente Supabase com JWT — RLS aplicado automaticamente; o engine nunca usa `service_role`
6. Testes unitarios: permissao negada (role insuficiente), rate limit excedido (mock contador), tool destrutiva retorna `needsConfirmation`, audit log gravado apos execucao bem-sucedida e apos falha

## Scope

### IN
- `packages/agent-harness/src/hooks/pre-tool.ts` (permissao, rate limit, destructive guard)
- `packages/agent-harness/src/hooks/post-tool.ts` (audit log)
- Integracao dos hooks no loop conversacional de 8.1
- Rate limiter em memoria (Map<userId, { count, windowStart }>) — suficiente para single-tenant MVP
- Sanitizacao de `input` antes de gravar no audit log (remover campos com "token", "password", "secret")
- Testes unitarios dos hooks

### OUT
- Rate limiter distribuido (Redis) — necessario apenas em multi-tenant (pos-MVP)
- Alert quando rate limit excedido frequentemente
- Dashboard de auditoria (pos-MVP — acesso direto ao banco por ora)
- Rotacao automatica de audit logs alem de 90 dias (pg_cron setup e responsabilidade de @devops)

## Technical Notes

### Pre-Tool Hook
```typescript
// packages/agent-harness/src/hooks/pre-tool.ts
interface PreToolResult {
  allowed: boolean
  needsConfirmation?: boolean
  denied?: boolean
  reason?: string
}

export const preToolHook = {
  async check(toolUse: ToolUseBlock, tool: UhuruTool, ctx: ToolContext): Promise<PreToolResult> {
    // 1. Role check
    const roleHierarchy = ['member', 'owner']
    const userLevel = roleHierarchy.indexOf(ctx.userRole)
    const requiredLevel = roleHierarchy.indexOf(tool.permissions.requiredRole)
    if (userLevel < requiredLevel) {
      return { allowed: false, denied: true, reason: `Tool '${tool.name}' requer role '${tool.permissions.requiredRole}'` }
    }

    // 2. Rate limit (30 msg/min por usuario)
    const key = ctx.userId
    const now = Date.now()
    const window = rateLimitStore.get(key) ?? { count: 0, windowStart: now }
    if (now - window.windowStart > 60_000) {
      // Nova janela
      rateLimitStore.set(key, { count: 1, windowStart: now })
    } else if (window.count >= 30) {
      return { allowed: false, denied: true, reason: 'Rate limit excedido (30 msg/min)' }
    } else {
      rateLimitStore.set(key, { ...window, count: window.count + 1 })
    }

    // 3. Destructive guard
    if (tool.permissions.isDestructive) {
      return { allowed: false, needsConfirmation: true }
    }

    return { allowed: true }
  }
}

// Rate limit store: Map em memoria (single-tenant MVP)
const rateLimitStore = new Map<string, { count: number; windowStart: number }>()
```

### Post-Tool Hook
```typescript
// packages/agent-harness/src/hooks/post-tool.ts
export const postToolHook = {
  async log(params: {
    sessionId: string
    toolName: string
    input: unknown
    output: unknown
    durationMs: number
    userId: string
    supabase: SupabaseClient<Database>
  }) {
    const sanitizedInput = sanitize(params.input)  // remove campos sensíveis
    await params.supabase.from('agent_audit_logs').insert({
      session_id: params.sessionId,
      tool_name: params.toolName,
      input: sanitizedInput,
      output: params.output,
      duration_ms: params.durationMs,
      user_id: params.userId
    })
  }
}
```

### Sanitizacao de Audit Log
```typescript
const SENSITIVE_KEYS = ['token', 'password', 'secret', 'key', 'auth', 'credential']
function sanitize(obj: unknown): unknown {
  if (typeof obj !== 'object' || obj === null) return obj
  return Object.fromEntries(
    Object.entries(obj as Record<string, unknown>).map(([k, v]) => [
      k,
      SENSITIVE_KEYS.some(s => k.toLowerCase().includes(s)) ? '[REDACTED]' : sanitize(v)
    ])
  )
}
```

### RLS (agent_audit_logs)
```sql
-- SELECT: owner vê todos; member vê apenas os próprios
CREATE POLICY "audit_select" ON public.agent_audit_logs FOR SELECT USING (
  public.get_user_role() = 'owner' OR user_id = auth.uid()
);
-- INSERT: apenas via service (nenhuma policy de INSERT para usuarios diretos)
-- Logs inseridos pelo post-hook com o cliente Supabase do usuário (JWT do usuário)
-- Como o post-hook usa o cliente com JWT do usuario, o RLS garante que o user_id = auth.uid()
```
