# Story 3.1 — Meta Marketing API — Token Management and Campaign Listing

**Epic:** 3 — Campaigns Integration (Meta + Google Ads)
**Status:** Draft
**Points:** 8

## Description

As a user, I want to connect my Meta Ads account and see my campaigns with real metrics, so that I can monitor Meta ad performance from Uhuru OS.

## Acceptance Criteria

1. Tabela `ad_platform_tokens` criada via migration: `id` (uuid PK), `client_id` (uuid NOT NULL FK clients(id) ON DELETE CASCADE), `platform` (text NOT NULL CHECK IN ('meta','google')), `access_token` (text NOT NULL — encrypted via pgcrypto/Vault), `refresh_token` (text NULLABLE), `expires_at` (timestamptz NULLABLE), `created_at` (timestamptz), `updated_at` (timestamptz); UNIQUE: `(client_id, platform)`
2. Fluxo de configuracao: usuario insere token de longa duracao do Meta para um cliente via formulario na tela de detalhe do cliente; token salvo encriptado
3. `GET /api/v1/clients/:id/campaigns?platform=meta` retorna lista de campanhas com metricas: spend, impressions, clicks, ctr, cpc, roas, status
4. Service layer chama Meta Marketing API (`/{ad_account_id}/campaigns` e `/{campaign_id}/insights`) com o token do cliente, retry com exponential backoff em erros 5xx e rate limit
5. Cache de metricas: resultados salvos na tabela `campaign_metrics_cache` com TTL de 30 minutos; se cache valido, retorna cache sem chamar a API externa
6. UI: tab "Campanhas" (ou secao) na tela de campanhas com filtro por cliente e plataforma; tabela com nome, status, spend, ROAS, CTR, CPC; indicador de quando cache foi atualizado

## Scope

### IN
- Migration `ad_platform_tokens` com RLS (owner only para write)
- Migration `campaign_metrics_cache` com indices
- Service: `campaign.service.ts` com Meta API client
- Logica de cache (upsert em `campaign_metrics_cache`, check TTL)
- Rotas de token: `POST /api/v1/clients/:id/tokens`, `DELETE /api/v1/clients/:id/tokens/:platform`, `GET /api/v1/clients/:id/tokens/status`
- UI de configuracao do token Meta e listagem de campanhas

### OUT
- Google Ads (Story 3.2)
- Alertas automaticos (Story 3.3)
- Dashboard (Epic 4 — consome dados desta story)
- Background job de refresh automatico (implementar junto com Story 3.3)

## Technical Notes

### Meta Marketing API Endpoints
```
GET /{ad_account_id}/campaigns
  ?fields=id,name,status,daily_budget,effective_status
  &access_token={token}

GET /{campaign_id}/insights
  ?fields=spend,impressions,clicks,ctr,cpc,actions
  &date_preset=last_30d
  &access_token={token}
```

### Cache Strategy
```typescript
// TTL check
const CACHE_TTL_MINUTES = 30
const isStale = (fetched_at: Date) =>
  Date.now() - fetched_at.getTime() > CACHE_TTL_MINUTES * 60 * 1000

// Upsert no cache
await supabase.from('campaign_metrics_cache').upsert({
  client_id, platform: 'meta', campaign_id, campaign_name,
  campaign_status, date: today, spend, impressions, clicks,
  ctr, cpc, conversions: 0, revenue: 0, roas, fetched_at: new Date()
}, { onConflict: 'client_id,platform,campaign_id,date' })
```

### Encriptacao do Token
- Usar `pgcrypto` (disponivel no Supabase): `encode(encrypt(token::bytea, key::bytea, 'aes'), 'base64')`
- Alternativa: Supabase Vault (mais simples para MVP)
- Key derivada de env var `TOKEN_ENCRYPTION_KEY`

### RLS (ad_platform_tokens)
```sql
ALTER TABLE public.ad_platform_tokens ENABLE ROW LEVEL SECURITY;
-- SELECT: mesmo acesso que cliente pai
-- INSERT/UPDATE/DELETE: owner only
CREATE POLICY "tokens_owner_write" ON public.ad_platform_tokens
  FOR ALL USING (public.get_user_role() = 'owner');
CREATE POLICY "tokens_select" ON public.ad_platform_tokens
  FOR SELECT USING (
    public.get_user_role() = 'owner' OR
    client_id IN (SELECT id FROM public.clients WHERE created_by = auth.uid())
  );
```
